BACKEND — Public Profile Visibility API (UPDATED)
=====================================================
Date: 2026-06-17

ENDPOINT: POST /api/user/visibility
  Authorization: Bearer <token>
  Body:
    {
      "visibility": {
        "merged": true/false,
        "workouts": true/false,
        "nutrition": true/false,
        "water": true/false,
        "statsBar": true/false,
        "leaderboard": true/false,
        "recentWorkouts": true/false,
        "publicLeaderboard": true/false
      }
    }
  → { "ok": true, "visibility": { ... } }

CRITICAL — Must persist these values per-user and return them in
/api/portal, /api/account/portal, and /api/user/profile responses
under one of these keys:

  root.publicVisibility
  root.publicProfileVisibility       (camelCase — dashboard checks first)
  root.profile.publicVisibility

Example portal response snippet:
{
  "publicProfileVisibility": {
    "merged": false,
    "workouts": false,
    "nutrition": false,
    "water": false,
    "statsBar": false,
    "leaderboard": false,
    "recentWorkouts": false,
    "publicLeaderboard": false
  }
}

The frontend checks `visibility.xxx === true`. Missing keys or undefined
means HIDDEN. ALL DEFAULTS MUST BE FALSE (max privacy).

The frontend now also caches visibility in localStorage as a fallback
(key: "tracker.publicVisibility"), but the backend is the authoritative
source. On dashboard refresh, the /api/portal response must include the
saved visibility so the checkboxes show the current state.
