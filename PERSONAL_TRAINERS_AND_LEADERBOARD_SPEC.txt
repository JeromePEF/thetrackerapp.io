================================================================================
PERSONAL TRAINERS · RUN CLUBS · GROUPS · PUBLIC LEADERBOARD
Backend integration spec for the new dashboard sections + leaderboard/profile pages
================================================================================

Last updated: 2026-05-22

The frontend is being extended with three concepts that need backend support:

  1. PERSONAL TRAINERS  – approve-based 1:N coach <-> client links, PT-only code
                         redemption, "become a PT" application flow.
  2. GROUPS + RUN CLUBS – memberships shown on every user's dashboard and
                         profile. (Telegram / iMessage / Run-Club types.)
  3. PUBLIC LEADERBOARD – a public /leaderboard page with category filters,
                         a click-through public profile at /u/<username>.

All endpoints below live on api.thetrackerapp.io and require
`Authorization: Bearer <session>` UNLESS marked `[public]`.

================================================================================
PART A · PERSONAL TRAINERS
================================================================================

ROLES
-----
- "Athlete"          – default role, can be coached by 0 or 1 PTs at a time.
- "Personal Trainer" – approved by site admin; gets a unique `trainerCode`
                       (e.g. `RIVER-9X4P`). Can be coaching N athletes.

A user can be BOTH an athlete and a PT (a PT may also have a coach).

DATA MODEL THE FRONTEND EXPECTS
-------------------------------
On every authenticated profile response (e.g. `/api/account/profile`):

```jsonc
{
  // ... existing fields ...

  "isPersonalTrainer":     false,            // approved-as-PT flag
  "personalTrainerStatus": "none",           // see status enum below
  "personalTrainer": null,                   // when user IS coached by a PT
  //                                        {
  //   "id":            "user_abc",
  //   "username":      "coachJay",
  //   "displayName":   "Coach Jay",
  //   "avatar":        "https://.../avatar.png",
  //   "since":         "2026-04-12",
  //   "trainerCode":   "RIVER-9X4P"
  // }
  "trainerCode":           "RIVER-9X4P",     // present when isPersonalTrainer
  "trainerCodePending":    null,             // when an athlete entered a code
                                              // but the PT hasn't approved yet:
                                              // { "trainerUsername":"coachJay",
                                              //   "submittedAt":"2026-05-22T01:23Z" }
  "trainerApplicationStatus":"none",         // PT-application status (see enum)
  "clientsAsTrainer": [],                    // present when isPersonalTrainer
  //   [
  //     {
  //       "id": "user_xyz",
  //       "username": "fierylion",
  //       "displayName": "🔥 Fierylion",
  //       "since": "2026-04-20",
  //       "lastLog": "2026-05-22",
  //       "status": "active"          // active | pending | removed
  //     }
  //   ]
  "trainerSettings": {                       // PT-owned settings (only on PT):
    "publicProfile": true,
    "acceptingClients": true,
    "bio": "10 yrs NSCA, hybrid lifter."
  }
}
```

ENUMS
-----
`personalTrainerStatus`:
  "none"                – user has no coach and no pending request
  "pending"             – user submitted a trainerCode, waiting for PT approval
  "linked"              – PT approved; `personalTrainer` is populated
  "rejected"            – PT rejected the request (frontend offers retry)
  "removed"             – previously linked, then unlinked by either party

`trainerApplicationStatus`:
  "none"                – not applied
  "pending"             – application submitted, awaiting site-admin review
  "approved"            – application approved; `isPersonalTrainer:true`,
                          `trainerCode` now present
  "rejected"            – application rejected (frontend can show note + retry)

ENDPOINTS
---------

[auth] GET /api/account/profile
  Already exists; just add the fields listed in DATA MODEL above.

[auth] POST /api/account/trainer-code/redeem
  Body: { "code": "RIVER-9X4P" }
  Behaviour:
    - Look up the PT whose `trainerCode` matches (case-insensitive, hyphen-flex).
    - If not found → 404 { ok:false, error:"Code not found" }.
    - If user already has a `linked` PT → 409 { ok:false, error:"Already linked" }.
    - Otherwise create a link request:
        { athleteId, trainerId, status:"pending", submittedAt:<now> }
      and notify the PT (text + push). Set the athlete's
      `personalTrainerStatus = "pending"` and `trainerCodePending`.
    - Returns the updated profile slice:
        { ok:true, personalTrainerStatus:"pending",
          trainerCodePending:{ trainerUsername, submittedAt } }

[auth] POST /api/account/trainer-code/cancel
  Body: {}
  Behaviour: athlete can withdraw their pending request before the PT acts.
  Returns: { ok:true, personalTrainerStatus:"none", trainerCodePending:null }

[auth] POST /api/trainer/requests/:athleteId/approve   (PT-only)
  Behaviour:
    - PT confirms the pending link.
    - Sets the athlete's `personalTrainerStatus="linked"` and populates
      `personalTrainer={…PT info…}`. PT's `clientsAsTrainer` gains the entry.
  Returns: { ok:true, client:{ id, username, displayName, since:<today> } }

[auth] POST /api/trainer/requests/:athleteId/reject    (PT-only)
  Body: { "reason"?: "string" }
  Sets athlete's `personalTrainerStatus="rejected"` and clears pending state.
  Returns: { ok:true }

[auth] POST /api/trainer/clients/:athleteId/remove     (PT-only OR self)
  Either party may break an existing link. Sets `personalTrainerStatus="removed"`.

[auth] GET  /api/trainer/requests   (PT-only)
  Returns pending requests waiting for this PT:
    { ok:true, requests:[
        { athleteId, username, displayName, submittedAt, avatar }
    ] }

[auth] POST /api/trainer/application
  Body:
    {
      "fullName":  "River Park",
      "bio":       "10 yrs NSCA, hybrid lifter.",
      "credentials":["NSCA-CPT","RDN"],
      "experienceYears": 10,
      "specialties":["Hybrid","Strength","Endurance"],
      "portfolioUrl":"https://...",
      "agreeTerms": true
    }
  Creates a pending application that an admin must approve out of band.
  Returns: { ok:true, trainerApplicationStatus:"pending" }

[auth] POST /api/trainer/application/withdraw
  Cancels a pending application. Returns { ok:true, trainerApplicationStatus:"none" }.

[admin] POST /api/admin/trainer/applications/:id/approve
[admin] POST /api/admin/trainer/applications/:id/reject
  Admin-only endpoints to flip a user to `isPersonalTrainer:true` and mint
  their `trainerCode`. Not consumed by the public frontend.

PART B · GROUPS, RUN CLUBS, COMMUNITIES
================================================================================

The frontend treats Telegram chat groups, iMessage group chats, and Run Clubs
uniformly – they all show up in a "Communities" rail on the dashboard and
profile page.

DATA MODEL ON /api/account/profile
----------------------------------

```jsonc
{
  "communities": [
    {
      "id":          "tg_morning_lifters",
      "kind":        "telegram",      // telegram | imessage | run-club | bracket
      "name":        "Morning Lifters",
      "emoji":       "🏋️",
      "memberCount": 156,
      "role":        "member",        // member | admin | owner
      "since":       "2026-02-10",
      "url":         "https://t.me/morninglifters",
      "isPublic":    true,
      "location":    null             // run-clubs only: "Austin, TX"
    },
    {
      "id":          "rc_austin",
      "kind":        "run-club",
      "name":        "Austin Sunrise Runners",
      "emoji":       "🏃",
      "memberCount": 42,
      "role":        "member",
      "since":       "2026-03-05",
      "url":         "/run-clubs/austin-sunrise",
      "isPublic":    true,
      "location":    "Austin, TX"
    }
  ]
}
```

ENDPOINTS
---------

[auth] GET /api/account/profile  – includes `communities` array (above).

[auth] POST /api/communities/:id/leave        – self-leave a community.
[auth] POST /api/communities/:id/join         – self-join (public only).
[auth] POST /api/communities/:id/request-join – request invite (private groups).

The existing /api/groups + /api/run-clubs endpoints continue to power their
discovery pages.

PART C · PUBLIC LEADERBOARD PAGE + USER PROFILES
================================================================================

The new /leaderboard page is public (works for signed-out visitors too). Click
a username and you land on /u/<username> which is also public unless the user
has flipped privacy off.

PAGE URLs
---------
  /leaderboard                         – page is already routed; HTML new
  /u/<username>                        – via Vercel rewrite to /u.html?u=<username>

  Lightweight existence check used by the dashboard "is this username taken"
  validators. Returns { ok:true, exists:true }.

FLAIRS  (tagpro-style)
----------------------
Flairs are user decorations earned by hitting milestones or winning brackets.
The frontend reads them from `user.flair` and renders:

  - `primary`   – short text shown as a coloured pill under the username.
  - `secondary` – optional second pill.
  - `color`     – hex used for the pill background tint.
  - `rarity`    – common | uncommon | rare | epic | legendary.
                  Maps to a CSS class for glow/animation strength.
  - `badges[]`  – earned medals; rendered as a grid on the profile page and
                  the highest-rarity 3 are shown next to the leaderboard row.

Suggested badge ids (anything else is fine, frontend just renders label/emoji):

  streak-7, streak-30, streak-100, streak-365,
  bracket-q1-2026-champion, bracket-q2-2026-champion, ...,
  bench-2pl, bench-3pl, bench-4pl,
  squat-3pl, squat-4pl, squat-5pl,
  deadlift-3pl, deadlift-4pl, deadlift-5pl,
  marathon-finisher, ultra-finisher,
  100-workouts, 365-workouts, 1000-workouts,
  founding-member, beta-tester,
  pt-approved, top-coach-2026

PART D · TOUCH POINTS THE FRONTEND ALREADY EXPECTS
================================================================================

These changes need to flow through endpoints we already use:

1. `/api/account/profile` (GET + PATCH)
   - GET adds: isPersonalTrainer, personalTrainerStatus, personalTrainer,
               trainerCode, trainerCodePending, trainerApplicationStatus,
               clientsAsTrainer, trainerSettings, communities, flair, ranks.
   - PATCH accepts any subset of these on PT-side updates:
       trainerSettings.publicProfile (bool)
       trainerSettings.acceptingClients (bool)
       trainerSettings.bio (string)

2. `/api/chart/data` (already documented)
   - No change required for PT/leaderboard, but we'll start sending
     `&forUser=<otherUsername>` for the profile page's sparklines:
       GET /api/chart/data?contact=<viewer>&forUser=<profile>&range=30d
     The backend should authorize this: a viewer can request charts for
     another user only if that user is public OR is one of their clients
     (when viewer is a PT) OR is their PT.

3. `/api/public-leaderboard`
   - No change needed for homepage panels; we keep them on the existing
     `/api/public-site-snapshot` upstream.

PART E · NOTIFICATIONS (out-of-band suggestions)
================================================================================

Whenever any of the following happen the backend should fire a text message
(via the user's preferred service – iMessage/SMS/Telegram):

  - Athlete redeems a PT code           → PT gets "X wants you to coach them.
                                          Reply YES to approve or NO to reject."
  - PT approves                          → Athlete gets "You're now linked with
                                          <PT> 🏋️ Tap to view their profile:
                                          https://thetrackerapp.io/u/<pt>"
  - PT rejects                           → Athlete gets a polite "Not a match
                                          this time — try another code anytime."
  - PT application approved              → User gets "You're a coach now! Your
                                          PT code is RIVER-9X4P. Share it and
                                          we'll text you when athletes redeem."
  - PT application rejected              → User gets reason + reapply CTA.
  - Bracket / streak badge earned        → Athlete + the PT both get a hype text.

================================================================================
PART G · TIMELINE & ROLLOUT
================================================================================

The frontend will ship a "soft" version first:

  v1 (now) – profile / leaderboard / dashboard PT panels render whatever the
             backend sends today (empty arrays + status:"none") so the new
             pages don't crash before the backend lands.
  v2       – wire up the PT redeem / apply flows once endpoints in PART A are
             reachable (the buttons will just POST and surface backend errors
             until they succeed).
  v3       – flairs + badges grid + sparklines once /api/u/:username returns
             the optional fields in PART C.

If any endpoint isn't ready, the frontend degrades gracefully (skips that
section instead of erroring).
