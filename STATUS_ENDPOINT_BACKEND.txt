CLOUDFLARE WAF — Backend action required (URGENT)
================================================================================

SUMMARY
-------
Cloudflare WAF / Bot Fight Mode is returning HTTP 403 to every
server-to-server request hitting api.thetrackerapp.io. This breaks
THREE production paths right now:

  1. /status page → fetches /api/status from the BROWSER. Returns 403.
     User sees "Status data temporarily unavailable" banner.

  2. /api/control → the Vercel serverless proxy at thetrackerapp.io/
     api/control tries to fetch https://api.thetrackerapp.io/control
     from the SERVER (Vercel). Gets 403. Proxy falls back to safe-default
     flags, and the WHOLE main page renders with everything hidden by
     default — users see nav links / sections / footer entries flipped
     OFF that were actually meant to be ON (or vice versa).

  3. /api/billing/stripe-prices → same WAF block from Vercel's IPs.
     Currently masked by the frontend's 15-min localStorage cache and
     STARTER_REGISTRY fallback, but once that expires, billing UI shows
     fallback prices instead of the live Stripe values.

Verified with curl from outside Cloudflare (and reproduced with
browser-shaped User-Agent + Sec-Fetch-* headers + Origin):

   $ curl -sS -o /dev/null -w "%{http_code}\n" \
       -H "Origin: https://thetrackerapp.io" \
       https://api.thetrackerapp.io/api/status            → 403
   $ curl -sS -o /dev/null -w "%{http_code}\n" \
       https://api.thetrackerapp.io/control               → 403
   $ curl -sS -o /dev/null -w "%{http_code}\n" \
       https://api.thetrackerapp.io/api/billing/stripe-prices  → 403

Every request returns the Cloudflare "Sorry, you have been blocked" HTML
interstitial. That's the symptom of Bot Fight Mode or Super Bot Fight
Mode treating outside-of-browser callers as "Likely Automated" with the
"Challenge" action that pure JSON clients can't solve.


REPRODUCING THE FAILURE
-----------------------
   $ curl -sS -o /dev/null -w "%{http_code}\n" \
       -H "Origin: https://thetrackerapp.io" \
       -H "Accept: application/json" \
       https://api.thetrackerapp.io/api/status

   → 403   (Cloudflare "Sorry, you have been blocked" HTML response)

Browser-shaped requests (User-Agent + Sec-Fetch-* headers + Origin) hit
the same 403. CORS preflight (OPTIONS) also returns 403, so even when a
browser DOES negotiate the request it can't get past WAF.

The same WAF policy is currently blocking /api/billing/stripe-prices
and /api/control too, but those endpoints have client caching so the
visible impact is delayed. /api/status is fetched fresh every minute so
the block surfaces immediately.


WHAT THE FRONTEND IS DOING (no changes needed here)
---------------------------------------------------
   GET https://api.thetrackerapp.io/api/status
       Accept: application/json
       Origin:  https://thetrackerapp.io
       Cache:   no-store

   On non-200 OR on network failure, the page shows:
       "Status data temporarily unavailable"   (banner)
       components grid renders an empty state with the support email

This is the safe fallback per LEGAL_AND_UX_POLISH spec — we DO NOT
pretend everything is operational when we can't get a response.


WHAT BACKEND NEEDS TO DO
------------------------

Option A — RECOMMENDED — One Cloudflare Custom Rule covering all four
public endpoints. Takes 2 minutes.

   1. Cloudflare Dashboard → Security → WAF → Custom Rules → Create rule.
   2. Rule expression (use the Edit-Expression view):

        (starts_with(http.request.uri.path, "/api/status"))
        or (starts_with(http.request.uri.path, "/api/control"))
        or (starts_with(http.request.uri.path, "/control"))
        or (starts_with(http.request.uri.path, "/api/billing/"))

   3. Action: **Skip**
        ✓ All remaining custom rules
        ✓ Super Bot Fight Mode
        ✓ Bot Fight Mode

   4. Save → Deploy. Applies within ~30 seconds.

   Verify with curl from outside Cloudflare:

        curl -i -H "Origin: https://thetrackerapp.io" \
             https://api.thetrackerapp.io/api/status
        curl -i -H "Origin: https://thetrackerapp.io" \
             https://api.thetrackerapp.io/control
        curl -i -H "Origin: https://thetrackerapp.io" \
             https://api.thetrackerapp.io/api/billing/stripe-prices

   Expect each: HTTP/2 200 + JSON body + CORS headers.

   While verifying, also confirm the matching Vercel routes are healthy:

        curl -i "https://thetrackerapp.io/api/control?nocache=$(date +%s)" \
             | grep -i x-control-source
        # Want:   x-control-source: upstream
        # Have:   x-control-source: fallback    ← current state

Option B — Move the four public endpoints behind a sub-path that has
its own WAF policy (e.g. "/public/*") and apply a relaxed managed
ruleset there. Same end result, more configuration.

Option C — Disable Bot Fight Mode entirely on api.thetrackerapp.io.
Not recommended — it's a useful layer for authed endpoints.

REQUIRED HEADERS
----------------
   Content-Type:                 application/json
   Access-Control-Allow-Origin:  *        (or echo Origin)
   Access-Control-Allow-Methods: GET, OPTIONS
   Cache-Control:                public, max-age=30
   Vary:                         Origin

ACCEPTABLE STATUS CODES
-----------------------
   200  → render normally
   503  → frontend will still render the response (use this for genuine
          system-wide outage when you want users to see the response
          AND know the status endpoint itself is degraded)
   anything else (including 403) → frontend shows fallback "temporarily
                                    unavailable"


FRONTEND MITIGATIONS ALREADY SHIPPED
------------------------------------
While backend fixes the WAF, the frontend now degrades safely:

   1. /api/control proxy fallback flipped to SAFE-BY-DEFAULT: every gated
      nav link, section, footer entry, and dashboard tab is FALSE when
      upstream is unreachable. Combined with the "show only on
      enabled === true" rule, the page renders with only always-visible
      UI (Pricing, leaderboards, the hero, iPhone mockup) rather than
      ALL gated items flashing visible.

   2. After receiving a fallback response from the proxy, the browser
      tries a direct fetch to https://api.thetrackerapp.io/control. Real
      browsers usually clear Cloudflare's bot challenge with cookies +
      browser UA, so users get the actual flags even when Vercel's
      server-side fetch is blocked. This is a workaround, not a fix —
      please add the WAF rule below so the proxy is the source of truth
      again.

   3. /status page banner reads "Status data temporarily unavailable"
      and the components grid renders empty (no false "all operational"
      lie). Re-fetches every 60s + on window focus.

So as soon as the WAF rule lands, every open browser session picks up
the real flags within ≤60 s with no reload.
